# 23-集合与文件夹级配置设计

> 本文档定义集合（Collection）和文件夹（Folder）级配置的继承模型、合并策略、脚本执行链、变量优先级、UI 方案和任务分解。所有请求发送时必须沿层级收集配置并合并。

---

## 1. 问题背景

### 1.1 现状缺陷

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | 集合/文件夹配置无 UI | 🔴 HIGH | `CollectionConfig` / `FolderConfig` 数据模型和持久化已有，但无 UI 编辑入口 |
| 2 | 变量优先级未实现 | 🔴 HIGH | `VariableScope` 定义了 5 层优先级，但 `sendRequest()` 仅填充 `global` + `environment`，`collection` 层永远为空 |
| 3 | Headers 不继承 | 🔴 HIGH | 集合/文件夹级 headers 不会合并到请求中 |
| 4 | Auth 不继承 | 🔴 HIGH | 集合/文件夹级 auth 不沿层级查找 |
| 5 | Scripts 不继承 | 🔴 HIGH | 集合/文件夹的 pre-request/post-response 脚本不执行 |
| 6 | `collectionVariables` 不传递 | 🟡 MED | `ScriptContext.collectionVariables` 字段存在但从不填充 |
| 7 | 脚本失败不中断请求 | 🟡 MED | Pre-request 脚本执行失败后请求仍会发送 |
| 8 | Auth 类型不一致 | 🟡 MED | `collection.rs::SavedAuth` 是松散 `{ auth_type, config: Value }`，`http.rs::AuthConfig` 是严格 tagged enum |

### 1.2 已有基础设施

- **Rust**: `CollectionConfig`（headers/auth/variables/scripts）、`FolderConfig`（同）已定义并持久化到 JSON
- **TypeScript**: `CollectionConfig`/`FolderConfig` 类型已有
- **Store**: `collection-store.ts` 已实现 `treeToIpcItems`/`ipcItemsToTree` 双向转换（含 config 字段映射）
- **Resolver**: `VariableScope` 接口已定义 `local/data/environment/collection/global` 五层
- **Script**: `ScriptContext` 已有 `collectionVariables` 字段

## 2. 配置继承模型

### 2.1 层级结构

```
Global Variables
  └── Collection Config
        ├── Variables
        ├── Headers
        ├── Auth
        └── Scripts (pre-request / post-response)
              └── Folder Config (Level 1)
                    ├── Variables
                    ├── Headers
                    ├── Auth
                    └── Scripts
                          └── Folder Config (Level 2)
                                └── ...
                                      └── Request Config
                                            ├── Variables (URL params)
                                            ├── Headers
                                            ├── Auth
                                            ├── Body
                                            └── Scripts
```

### 2.2 合并策略

| 配置项 | 合并策略 | 详细规则 |
|--------|----------|----------|
| **Variables** | 层级覆盖 | 同 key 时高层覆盖低层。优先级：`local > data > request > folder(深层>浅层) > collection > environment > global` |
| **Headers** | 追加 + 覆盖 | 集合 headers 全部追加 → 文件夹 headers 追加（同 key 覆盖较低层） → 请求 headers 追加（同 key 覆盖所有上层） |
| **Auth** | 最具体优先 | 请求 auth ≠ none → 用请求的；请求 auth = none → 沿层向上找第一个 auth ≠ none 的；全为 none → 无认证 |
| **Scripts** | 链式执行 | Pre-request: Collection → Folder(浅→深) → Request；Post-response: Request → Folder(深→浅) → Collection |
| **Settings** | 最近覆盖 | 请求有自定义 settings 用请求的，否则沿层向上找第一个非默认的 settings |

### 2.3 合并示例

**场景**：集合 "Users API" → 文件夹 "Admin" → 请求 "Delete User"

```
Collection Config:
  headers: [Authorization: Bearer {{token}}]
  auth: none (使用 headers 中的 token)
  variables: [base_url: https://api.example.com, version: v2]
  scripts.pre-request: console.log("Collection pre")

Folder "Admin" Config:
  headers: [X-Admin: true]
  auth: none
  variables: [admin_role: superadmin]
  scripts.pre-request: console.log("Admin folder pre")

Request "Delete User":
  headers: []
  auth: { type: "basic", config: { username: "admin", password: "pass" } }
  scripts.pre-request: console.log("Request pre")

合并结果：
  有效 Headers: [Authorization: Bearer {{token}}, X-Admin: true]
  有效 Auth: Basic (请求级覆盖了集合的 header 方案)
  有效 Variables: { base_url: "https://api.example.com", version: "v2", admin_role: "superadmin" }
  执行顺序 (pre-request): Collection → Admin → Request
```

## 3. 变量优先级

### 3.1 完整优先级表

| 优先级 | 层级 | 来源 | 说明 |
|--------|------|------|------|
| 1 (最高) | `local` | 脚本中 `pm.variables.set()` | 运行时动态设置 |
| 2 | `data` | 数据驱动变量 | 预留，暂未使用 |
| 3 | `request` | 请求 URL params 中的变量 | 请求级覆盖 |
| 4 | `folder` | 当前文件夹 + 父文件夹变量 | 深层文件夹覆盖浅层 |
| 5 | `collection` | 集合 `CollectionConfig.variables` | 集合级默认 |
| 6 | `environment` | 活跃环境变量 | 环境切换生效 |
| 7 (最低) | `global` | 全局变量 | 全局默认 |

### 3.2 环境变量 vs 集合变量的关系

**规则：`environment > collection`**

理由：
- 环境变量是用户主动切换的上下文（dev/staging/prod），语义上应覆盖集合的默认值
- 集合变量是 API 级别的默认配置，环境变量是部署级别的覆盖
- 与 Postman 行为一致

### 3.3 文件夹变量合并

文件夹变量按路径从浅到深合并，深层覆盖浅层：

```
Collection: { base_url: "https://api.example.com" }
  Folder L1 "v2": { base_url: "https://api.example.com/v2", timeout: "30" }
    Folder L2 "admin": { admin_role: "super" }

合并后: { base_url: "https://api.example.com/v2", timeout: "30", admin_role: "super" }
```

### 3.4 VariableScope 扩展

```typescript
interface VariableScope {
  local?: Record<string, string>;        // pm.variables.set()
  data?: Record<string, string>;         // 预留
  request?: Record<string, string>;      // 新增：请求级
  folder?: Record<string, string>;        // 新增：文件夹级（已合并深层覆盖浅层）
  collection?: Record<string, string>;   // 集合级
  environment?: Record<string, string>;  // 环境级
  global?: Record<string, string>;       // 全局级
}
```

### 3.5 解析流程

```
resolveVariable(key):
  1. local[key]          → 有则返回
  2. data[key]            → 有则返回
  3. request[key]         → 有则返回
  4. folder[key]          → 有则返回
  5. collection[key]      → 有则返回
  6. environment[key]     → 有则返回
  7. global[key]          → 有则返回
  8. 动态变量 ($guid 等)  → 有则返回
  9. 未解析              → 保留 {{key}} 原样，Console 记录 warn
```

## 4. 脚本执行链

### 4.1 Pre-request 执行流程

```
sendRequest() 被调用
  │
  ├── 1. 解析请求所属层级路径
  │     collectionId → [folderId L1, folderId L2, ...] → requestId
  │
  ├── 2. 收集各级 scripts
  │     collectionScripts = collection.config?.scripts
  │     folderScripts[0]  = folder_L1.config?.scripts
  │     folderScripts[1]  = folder_L2.config?.scripts
  │     requestScripts    = request.scripts
  │
  ├── 3. 链式执行 pre-request
  │     for script of [collectionScripts, ...folderScripts, requestScripts]:
  │       if script.preRequest:
  │         result = executeScript({ code: script.preRequest, context })
  │         if result.error:
  │           ❌ 中断请求，显示错误
  │         if result.modifiedRequest:
  │           合并修改到请求配置
  │
  ├── 4. 发送 HTTP 请求
  │
  └── 5. 链式执行 post-response（反序）
        for script of [requestScripts, ...folderScripts.reverse(), collectionScripts]:
          if script.postResponse:
            result = executeScript({ code: script.postResponse, context })
            if result.error:
              ⚠️ 记录错误到 Console，不中断后续脚本
```

### 4.2 中断行为定义

| 阶段 | 失败行为 | 用户反馈 |
|------|----------|----------|
| Collection pre-request | 中断，不发送请求 | Console: `[Pre-request][Collection] 执行失败: {error}` |
| Folder pre-request | 中断，不发送请求 | Console: `[Pre-request][Folder: {name}] 执行失败: {error}` |
| Request pre-request | 中断，不发送请求 | Console: `[Pre-request][Request] 执行失败: {error}` |
| Request post-response | 记录错误，不中断 | Console: `[Post-response][Request] 执行失败: {error}` |
| Folder post-response | 记录错误，不中断 | Console: `[Post-response][Folder: {name}] 执行失败: {error}` |
| Collection post-response | 记录错误，不中断 | Console: `[Post-response][Collection] 执行失败: {error}` |

### 4.3 中断时的 UI 状态

- **Response Panel**: 显示错误卡片而非正常响应
  - 标题: "Script Error"
  - 内容: 具体错误信息 + 执行阶段
  - 操作: "Edit Script" 按钮（跳转到对应脚本编辑器）
- **Console Panel**: 红色错误日志
- **Tab 状态**: 请求 Tab 标记为错误状态（可选：Tab 标题加 ⚠️ 前缀）

### 4.4 ScriptContext 扩展

```typescript
interface ScriptContext {
  request?: { method; url; headers; body };
  response?: { status; statusText; headers; body; time };
  environment?: Record<string, string>;
  collectionVariables?: Record<string, string>;  // 合并后的集合+文件夹变量
  globals?: Record<string, string>;
  folderPath?: string[];      // 新增：当前文件夹路径（供脚本判断层级）
  collectionName?: string;    // 新增：集合名称
```

## 6. 数据流改造

### 6.1 sendRequest() 完整流程

```
sendRequest(tabId, method, url)
  │
  ├── 1. 解析请求层级路径
  │     - 从 CollectionStore 查找请求所属的 collectionId
  │     - 构建 folderPath: [root_folder, ..., leaf_folder]
  │     - 提取各级 config
  │
  ├── 2. 合并 Variables
  │     - collectionVars = collection.config?.variables → Record<string, string>
  │     - folderVars = 沿 folderPath 从浅到深合并 → Record<string, string>
  │     - 构建 VariableScope:
  │       { global, environment, collection: collectionVars, folder: folderVars }
  │
  ├── 3. 合并 Headers
  │     - merged = []
  │     - merged.push(...collection.config?.headers)    // 集合级
  │     - merged.push(...folder.config?.headers)         // 文件夹级（覆盖同名 key）
  │     - merged.push(...request.headers)                // 请求级（覆盖同名 key）
  │     - 使用 resolver.resolve() 解析 {{variable}} 引用
  │
  ├── 4. 合并 Auth
  │     - 如果 request.auth.type !== "none" → 用请求的
  │     - 否则沿 folderPath 深→浅查找第一个 auth.type !== "none"
  │     - 否则用 collection.config?.auth
  │     - 否则无认证
  │
  ├── 5. 链式执行 Pre-request Scripts
  │     - 顺序: collection → folders(浅→深) → request
  │     - 每步构建 ScriptContext（含合并后的 collectionVariables）
  │     - 任一步失败 → 中断，显示 Script Error
  │     - modifiedRequest → 合并到请求配置
  │
  ├── 6. 发送 HTTP 请求（现有逻辑）
  │
  ├── 7. 链式执行 Post-response Scripts
  │     - 顺序: request → folders(深→浅) → collection
  │     - 每步构建 ScriptContext（含 response）
  │     - 失败 → 记录到 Console，不中断
  │
  └── 8. 更新 UI 状态
        - 成功: 正常响应
        - 脚本中断: 显示 Script Error 卡片
```

### 6.2 层级路径解析算法

```typescript
interface ResolvedHierarchy {
  collectionId: string;
  collectionConfig: CollectionConfig | undefined;
  folderPath: Array<{
    id: string;
    name: string;
    config: FolderConfig | undefined;
  }>;
  requestNode: CollectionTreeNode & { type: "request" };
}

function resolveRequestHierarchy(
  collections: CollectionItem[],
  requestId: string
): ResolvedHierarchy | null {
  for (const col of collections) {
    const result = findInTree(col.items, requestId, []);
    if (result) {
      return {
        collectionId: col.id,
        collectionConfig: col.config,
        folderPath: result.folderPath,
        requestNode: result.node,
      };
    }
  }
  return null;
}

function findInTree(
  items: CollectionTreeNode[],
  requestId: string,
  currentPath: Array<{ id: string; name: string; config: FolderConfig | undefined }>
): { node: CollectionTreeNode & { type: "request" }; folderPath: typeof currentPath } | null {
  for (const item of items) {
    if (item.type === "request" && item.id === requestId) {
      return { node: item, folderPath: currentPath };
    }
    if (item.type === "folder") {
      const result = findInTree(
        item.items,
        requestId,
        [...currentPath, { id: item.id, name: item.name, config: item.config }]
      );
      if (result) return result;
    }
  }
  return null;
}
```

### Phase A — 数据层 + 继承合并（Day 1-4）

- **依赖**：无
- **工时**：4h
- **验收标准**：
  - `CollectionStore` 新增 `resolveRequestHierarchy(requestId)` 方法
  - `VariableScope` 新增 `folder` 和 `request` 层级
  - `VariableResolver` 更新 7 层优先级解析
  - 单元测试覆盖：多层文件夹变量覆盖、环境变量覆盖集合变量
- **关键文件**：`collection-store.ts`, `resolver.ts`

#### A.2 — Headers/Auth 合并逻辑
- **依赖**：A.1
- **工时**：4h
- **验收标准**：
  - 实现 `mergeHeaders(hierarchy)` — 沿层级追加+覆盖
  - 实现 `resolveAuth(hierarchy)` — 沿层查找第一个非 none auth
  - `SavedAuth` 统一为 tagged enum（迁移策略：V2 优先，V1 fallback）
  - Rust 端新增 `update_collection_config` / `update_folder_config` commands
- **关键文件**：`collection-store.ts`, `collection.rs`

#### A.3 — 脚本执行链 + 中断逻辑
- **依赖**：A.1, A.2
- **工时**：6h
- **验收标准**：
  - 实现 `executePreRequestChain(hierarchy)` — 集合→文件夹→请求，失败中断
  - 实现 `executePostResponseChain(hierarchy)` — 请求→文件夹→集合，失败不中断
  - `ScriptContext` 新增 `collectionVariables`、`folderPath`、`collectionName`
  - 脚本失败时返回结构化错误：`{ phase: "pre-request", source: "Collection" | "Folder: {name}" | "Request", error: string }`
  - Console 记录每步执行日志
- **关键文件**：`request-store.ts`

#### A.4 — sendRequest() 完整改造
- **依赖**：A.1, A.2, A.3
- **工时**：4h
- **验收标准**：
  - `sendRequest()` 调用 `resolveRequestHierarchy()` 获取层级
  - 合并 variables → headers → auth → scripts
  - 链式执行 pre-request（中断）
  - 发送请求
  - 链式执行 post-response（非中断）

- **关键文件**：`request-store.ts`

#### C.1 — 请求 Auth/Headers 继承指示

- **工时**：4h
- **验收标准**：
  - 请求 Auth tab 新增 "Inherit from parent" 选项
  - 请求 Headers tab 显示继承的 headers（灰色只读）+ 来源标签
  - 点击继承 header 的来源标签 → 跳转到对应集合/文件夹设置 Tab
- **关键文件**：`RequestPanel.tsx`

#### C.2 — ScriptErrorCard + 错误状态
- **依赖**：A.4
- **工时**：4h
- **验收标准**：
  - `ScriptErrorCard` 组件：标题 + 错误信息 + 执行阶段 + "Edit Script" 按钮
  - Response Panel 在脚本中断时渲染 `ScriptErrorCard`
  - Console 记录完整执行链日志
  - "Edit Script" 按钮 → 跳转到对应脚本编辑位置
- **关键文件**：`ScriptErrorCard.tsx`, `ResponsePanel.tsx`

#### C.3 — 端到端联调
- **依赖**：A.4, B.4, C.1, C.2
- **工时**：6h
- **验收标准**：
  - **变量继承**：集合变量 → 文件夹变量覆盖 → 环境变量覆盖 → 请求中 `{{variable}}` 正确解析
  - **Headers 继承**：集合 headers + 文件夹 headers + 请求 headers 合并正确
  - **Auth 继承**：请求 none → 继承文件夹 auth → 继承集合 auth
  - **Scripts 链式执行**：Collection pre → Folder pre → Request pre → 发送 → Request post → Folder post → Collection post
  - **脚本中断**：Collection pre-request 失败 → 请求不发送 → ScriptErrorCard 显示
  - **脚本非中断**：Post-response 失败 → 请求响应正常显示 → Console 记录错误
  - **设置 UI**：双击集合 → 打开设置 Tab → 编辑变量/headers/auth/scripts → 保存 → 下次请求生效
  - **继承预览**：选择请求 → 显示有效变量及来源层级
  - **Tab 去重**：同一集合/文件夹不重复打开 Tab
  - 全流程无 console error，TypeScript 类型检查通过，ESLint 无 warning
- **关键文件**：所有相关文件
