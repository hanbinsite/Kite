# 状态持久化与变量同步策略

> 本文档定义前端 Zustand Store 与 Rust 后端 Storage 之间的数据同步协议，包括读写时机、冲突解决策略、变量变更传播机制和脏数据检测。

---

## 1. 存储架构总览

### 1.1 双层存储模型

```
┌───────────────────────────────────────────────────────────┐
│                     Frontend (WebView)                     │
│  ┌─────────┐ ┌─────────┐ ┌──────────┐ ┌───────────────┐  │
│  │ UIStore │ │TabStore │ │RequestSt │ │EnvironmentSt  │  │
│  └────┬────┘ └────┬────┘ └─────┬────┘ └───────┬───────┘  │
│       │           │            │               │           │
│       └───────────┴────────────┴───────────────┘           │
│                         │ Tauri IPC                         │
│                         │ invoke() / listen()               │
├─────────────────────────┼─────────────────────────────────┤
│                  Rust Backend                             │
│                         │                                  │
│       ┌─────────────────┴─────────────────┐               │
│       │              AppState              │               │
│       └──────┬──────────────┬─────────────┘               │
│              │              │                              │
│   ┌──────────▼──┐  ┌───────▼────────┐                     │
│   │ File System │  │    SQLite      │                     │
│   │ (Collections│  │ (History       │                     │
│   │  Environ-   │  │  Cookies       │                     │
│   │  ments)     │  │  Settings)     │                     │
│   └─────────────┘  └────────────────┘                     │
└───────────────────────────────────────────────────────────┘
```

### 1.2 存储职责划分

| 数据类型 | 存储位置 | 格式 | 读写方 | 原因 |
|---------|---------|------|--------|------|
| Collections | File System | JSON 目录树 | Rust | Git 友好、便于导入导出 |
| Environments | File System | JSON 文件 | Rust | 与集合同目录、版本控制 |
| History | SQLite | 表 `history` | Rust | 高频写入、需要索引查询 |
| Cookies | SQLite | 表 `cookie_jar` | Rust | 结构化查询、过期清理 |
| Settings | SQLite | 表 `settings` | Rust | KV 查询、全局唯一 |
| UI 偏好 | localStorage | JSON | 前端 | 轻量、无需 Rust |
| Tab 状态 | 内存 + localStorage | JSON | 前端 | 会话级、重启可恢复 |
| 请求草稿 | 内存 | Zustand | 前端 | 临时编辑态 |
| Vault Secrets | File System (加密) | .enc.json | Rust | 安全隔离 |

> **存储位置澄清**：环境变量以 **File System** 为权威存储（`environments/{id}.json`），Git 友好且可手动编辑。
> 脚本中 `pm.environment.set()` 修改变量时：立即写入 File System，同时更新前端 Store。
> SQLite 中不存储环境变量。仅 History、Cookies、Settings 使用 SQLite。

## 2. 读写时机协议

### 2.1 应用启动流程

```
App Launch
  │
  ├─ 1. Rust: AppState::new() → 初始化 SQLite, 执行 migrate()
  │     ├── 确保 schema_version 表存在
  │     ├── 创建/迁移 history, cookie_jar, settings 表
  │     └── 读取 settings 表 → 构建默认配置
  │
  ├─ 2. Rust: 扫描 {app_data}/collections/ → 加载集合目录列表
  │     ├── 仅读取顶层 collection.json (不递归加载请求)
  │     └── 返回 CollectionSummary[] (id, name, request_count, updated_at)
  │
  ├─ 3. Rust: 扫描 {app_data}/environments/ → 加载环境列表
  │     └── 返回 EnvironmentSummary[] (id, name, color, variable_count)
  │
  ├─ 4. Frontend: 从 localStorage 恢复 UI 偏好
  │     ├── theme, sidebar.visible, sidebar.width, splitRatio
  │     └── 上次打开的 Tab 列表 (tab_ids + activeTabId)
  │
  ├─ 5. Frontend: invoke('get_collections') → 填充 Sidebar
  │     └── 懒加载：仅展开的集合才加载请求列表
  │
  ├─ 6. Frontend: invoke('get_environments') → 填充环境选择器
  │     └── 仅加载当前环境的变量值
  │
  └─ 7. Frontend: invoke('get_recent_history', {limit: 50}) → 填充历史
```

### 2.2 请求发送流程中的读写

```
User Clicks Send
  │
  ├─ 1. Frontend: 构建 HttpRequestConfig (从 RequestStore + EnvironmentStore 合并)
  │     ├── 解析 {{variable}} → 替换为当前环境变量值
  │     ├── 合并 Auth 配置
  │     └── 合并 RequestSettings
  │
  ├─ 2. Frontend: invoke('send_http_request', { config })
  │
  ├─ 3. Rust: 执行 HTTP 请求
  │     ├── 读取 Cookie jar → 注入 Cookie 头
  │     └── 请求完成后 → 写入 Cookie jar
  │
  ├─ 4. Rust: 返回 HttpResponse
  │
  ├─ 5. Rust (异步): invoke('insert_history', { entry })
  │     └── 写入 SQLite history 表
  │
  └─ 6. Frontend: 更新 RequestStore.currentResponse
        ├── 更新 isLoading = false
        └── 更新 ResponsePanel
```

### 2.3 集合/环境编辑的读写

| 操作 | 写入时机 | 写入位置 | 前端更新 |
|------|---------|---------|---------|
| 创建集合 | 立即 | FS: `collections/{id}/collection.json` | Store 添加条目 |
| 重命名集合 | 立即 | FS: `collection.json` 的 name 字段 | Store 更新名称 |
| 删除集合 | 立即 | FS: 递归删除整个集合目录 | Store 移除条目 |
| 创建请求 | 立即 | FS: `requests/{id}.req.json` | Store 添加请求 |
| 编辑请求 | **延迟** | 内存中暂存，保存时写入 FS | Store 更新 (dirty) |
| 保存请求 | 用户 ⌘S 或切换 Tab 时 | FS: `requests/{id}.req.json` | Store 标记 clean |
| 创建环境 | 立即 | FS: `environments/{id}.json` | Store 添加条目 |
| 编辑环境变量 | **延迟** | 内存暂存，保存时写入 FS | Store 更新 (dirty) |
| 保存环境 | 用户 ⌘S 或关闭编辑器时 | FS: `environments/{id}.json` | Store 标记 clean |
| 修改设置 | 立即 | SQLite: settings 表 | 无需额外更新 |

## 3. 脏数据检测与保存策略

### 3.1 脏数据标记

```typescript
interface TabState {
  id: string;
  requestId: string | null;  // null = 临时请求
  isDirty: boolean;           // 编辑后未保存
  dirtyFields: Set<string>;   // 具体修改的字段
  lastSavedHash: string;      // 上次保存时的内容哈希
}
```

### 3.2 脏检测算法

```typescript
function computeHash(data: unknown): string {
  const json = JSON.stringify(data, Object.keys(data as object).sort());
  let hash = 0;
  for (let i = 0; i < json.length; i++) {
    const char = json.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash |= 0;
  }
  return hash.toString(36);
}

function checkDirty(tab: TabState, currentData: RequestData): boolean {
  return computeHash(currentData) !== tab.lastSavedHash;
}
```

### 3.3 自动保存策略

| 场景 | 行为 | 说明 |
|------|------|------|
| 用户 ⌘S | 立即保存 | 显式保存 |
| 切换 Tab | 自动保存当前 Tab | 避免数据丢失 |
| 关闭 Tab (dirty) | 弹出确认对话框 | "Save / Don't Save / Cancel" |
| 关闭窗口 | 逐个检查 dirty Tab | 依次弹出确认或全部保存 |
| 应用失焦 | 不自动保存 | 避免意外写入 |
| ⌘S 时网络断开 | 保存到内存 + 标记 pending | 下次连接时重试 |

### 3.4 保存防抖

```typescript
const SAVE_DEBOUNCE_MS = 1000;

const debouncedSave = debounce(async (tabId: string) => {
  const tab = useTabStore.getState().getTab(tabId);
  if (!tab?.isDirty) return;

  const requestData = useRequestStore.getState().getRequestData(tabId);
  await invoke('write_file', {
    path: `collections/${tab.collectionId}/requests/${tab.requestId}.req.json`,
    content: JSON.stringify(requestData, null, 2),
  });

  useTabStore.getState().markClean(tabId);
}, SAVE_DEBOUNCE_MS);
```

## 4. 变量变更传播机制

### 4.1 变更传播架构

```
变量变更源
  ├── 用户编辑环境变量 (UI)
  ├── 脚本执行 pm.variables.set() (Rust → Frontend)
  ├── 切换当前环境 (UI)
  ├── 导入新环境 (Rust → Frontend)
  └── Vault 解锁/锁定 (Rust → Frontend)
        │
        ▼
  EnvironmentStore.setVariable() / setActiveEnvironment()
        │
        ├── 1. 更新 Store 内部状态 (Zustand)
        │
        ├── 2. 广播 Tauri 事件 'environment-changed'
        │     payload: { type: 'variable_update' | 'environment_switch', ... }
        │
        └── 3. 所有活跃 Tab 的 URL/变量引用自动重解析
              (useEffect 监听 EnvironmentStore.activeVariables)
```

### 4.2 前端事件监听

```typescript
// packages/core/src/environment/sync.ts
import { listen } from "@tauri-apps/api/event";
import { useEnvironmentStore } from "../../apps/desktop/src/stores/environment-store";

interface EnvironmentChangedPayload {
  type: 'variable_update' | 'environment_switch' | 'environment_created' | 'environment_deleted';
  environmentId?: string;
  variableKey?: string;
  variableValue?: string;
}

export function initEnvironmentSync() {
  listen<EnvironmentChangedPayload>('environment-changed', (event) => {
    const store = useEnvironmentStore.getState();

    switch (event.payload.type) {
      case 'variable_update':
        store.updateVariableLocal(
          event.payload.environmentId!,
          event.payload.variableKey!,
          event.payload.variableValue!
        );
        break;

      case 'environment_switch':
        store.setActiveEnvironment(event.payload.environmentId!);
        break;

      case 'environment_created':
        store.addEnvironmentFromBackend(event.payload.environmentId!);
        break;

      case 'environment_deleted':
        store.removeEnvironmentFromBackend(event.payload.environmentId!);
        break;
    }
  });
}
```

### 4.3 Rust → 前端变量同步（脚本执行后）

```
脚本 pm.variables.set('token', 'abc')
  │
  ├─ 1. Rust: ScriptResult.variables = { "token": "abc" }
  │
  ├─ 2. Rust: 写入 File System (环境变量持久化)
  │     └── 原子写入 environments/{env_id}.json
  │
  ├─ 3. Rust: emit('environment-changed', { type: 'variable_update', ... })
  │
  └─ 4. Frontend: 收到事件 → EnvironmentStore.updateVariableLocal()
        └── 所有使用 {{token}} 的 URL/Headers/Body 自动重渲染
```

### 4.4 前端 → Rust 变量同步（用户编辑）

```
用户在环境编辑器修改变量值
  │
  ├─ 1. Frontend: EnvironmentStore.setVariable(key, value)
  │     └── 立即更新 Store（UI 无延迟）
  │
  ├─ 2. Frontend: invoke('set_environment_variable', { envId, key, value })
  │     └── 写入 FS: environments/{id}.json
  │
  └─ 3. Rust: 接收后原子写入文件
        └── 成功后无需 emit（前端已更新）
        └── 失败时 emit('environment-write-failed', { key, error })
              └── 前端回滚 Store 变更
```

## 5. 冲突解决策略

### 5.1 单用户场景（默认）

本项目 MVP 为单用户本地应用，**无多端同步冲突**。但需处理以下边界情况：

| 冲突场景 | 策略 | 说明 |
|---------|------|------|
| 前端保存时文件已被外部修改 | **Last Write Wins** + 用户通知 | 读取文件 mtime，若比预期新则弹出 "文件已被外部修改，是否覆盖？" |
| 两个 Tab 编辑同一请求 | **Tab 级隔离** | 每个 Tab 维护独立的编辑副本，保存时覆盖 |
| 脚本修改变量 vs 用户同时编辑 | **脚本优先** | 脚本执行结果立即生效，用户编辑区显示新值（可能覆盖用户输入） |
| Rust 写入失败 | **前端回滚** | invoke 返回错误后，前端回滚 Store 到之前的状态 |
| 大文件写入中应用崩溃 | **原子写入** | 先写临时文件，rename 替换原文件 |

### 5.2 原子写入实现

```rust
pub fn atomic_write(path: &Path, content: &str) -> Result<(), AppError> {
    let tmp_path = path.with_extension("tmp");

    fs::write(&tmp_path, content)
        .map_err(|e| AppError::StorageWriteFailed {
            path: path.to_string_lossy().to_string(),
            source: e.to_string(),
        })?;

    fs::rename(&tmp_path, path)
        .map_err(|e| {
            let _ = fs::remove_file(&tmp_path);
            AppError::StorageWriteFailed {
                path: path.to_string_lossy().to_string(),
                source: e.to_string(),
            }
        })?;

    Ok(())
}
```

### 5.3 文件外部修改检测

```typescript
// 保存前检查文件是否被外部修改
async function saveWithConflictCheck(
  filePath: string,
  content: string,
  expectedMtime: number
): Promise<SaveResult> {
  const currentMtime = await invoke<number>('get_file_mtime', { path: filePath });

  if (currentMtime > expectedMtime) {
    return { status: 'conflict', currentMtime };
  }

  await invoke('write_file', { path: filePath, content });
  return { status: 'saved' };
}
```

## 6. 数据一致性保证

### 6.1 写入顺序保证

所有对同一资源的写入操作在 Rust 端通过 `tokio::sync::Mutex` 串行化：

```rust
pub struct AppState {
    pub http_client: HttpClientState,
    pub db: Mutex<SqlitePool>,
    pub fs_lock: Mutex<()>,  // 文件系统写入锁
}

// 所有文件写入通过 fs_lock 串行化
async fn write_collection_file(
    state: &AppState,
    path: &str,
    content: &str,
) -> Result<(), AppError> {
    let _lock = state.fs_lock.lock().await;
    let full_path = Path::new(&state.data_dir).join(path);
    atomic_write(&full_path, content)
}
```

### 6.2 读取一致性

- 集合数据：前端启动时加载，之后通过 Tauri 事件通知变更（非轮询）
- 历史数据：每次打开历史面板时重新查询 SQLite（不缓存）
- 环境变量：切换环境时重新加载该环境的全部变量
- 设置数据：每次打开设置页面时重新查询 SQLite

### 6.3 数据校验

```rust
pub fn validate_collection_json(data: &serde_json::Value) -> Result<Collection, AppError> {
    let schema_version = data.get("schemaVersion")
        .and_then(|v| v.as_str())
        .unwrap_or("1");

    if schema_version != COLLECTION_SCHEMA_VERSION {
        return Err(AppError::StorageMigrationFailed {
            from: schema_version.to_string(),
            to: COLLECTION_SCHEMA_VERSION.to_string(),
        });
    }

    serde_json::from_value::<Collection>(data.clone())
        .map_err(|e| AppError::ImportParseError {
            detail: format!("Invalid collection format: {}", e),
        })
}
```

## 7. 数据备份与恢复

### 7.1 自动备份

| 数据 | 备份策略 | 保留 |
|------|---------|------|
| Collections | 每次保存时创建 `.bak` 文件 | 1 份（上一次的版本） |
| SQLite | 应用启动时复制为 `app.db.bak` | 1 份 |
| Settings | 不单独备份（包含在 SQLite 备份中） | — |

### 7.2 手动导出

通过设置页面的 Data Management 子页：

- **Export All Data**：将 collections/ + environments/ + globals 打包为 ZIP
- **Import Data**：从 ZIP 还原，支持选择性导入

### 7.3 数据恢复流程

```
应用启动 → SQLite 打开失败
  │
  ├─ 1. 检测 app.db 是否损坏 (PRAGMA integrity_check)
  │
  ├─ 2. 损坏 → 尝试从 app.db.bak 恢复
  │     ├── 恢复成功 → 提示用户 "Database recovered from backup"
  │     └── 恢复失败 → 创建新数据库 + 提示 "Database reset, collections preserved"
  │
  └─ 3. 集合文件损坏 → 跳过该集合 + 提示用户
        └── 集合目录有 .bak 文件 → 尝试从 .bak 恢复
```

## 8. 状态恢复（Tab 重启后）

### 8.1 localStorage 存储结构

```typescript
interface PersistedState {
  version: 1;
  ui: {
    theme: 'dark' | 'light' | 'system';
    sidebarVisible: boolean;
    sidebarWidth: number;
    splitRatio: number;
  };
  tabs: {
    openTabs: Array<{
      requestId: string | null;
      collectionId: string | null;
      url: string;
      method: string;
    }>;
    activeTabId: string | null;
  };
  activeEnvironmentId: string | null;
}
```

### 8.2 恢复流程

```
App 启动
  │
  ├─ 1. 从 localStorage 读取 PersistedState
  │
  ├─ 2. 恢复 UI 偏好 → useUIStore
  │
  ├─ 3. 恢复 Tab 列表 → useTabStore
  │     ├── 对于关联集合的 Tab: invoke('get_request', { id }) 加载完整数据
  │     ├── 对于临时请求 Tab: 仅恢复 method + URL (body 丢失，可接受)
  │     └── 无效 Tab (请求已被删除): 跳过
  │
  ├─ 4. 恢复活跃环境 → useEnvironmentStore
  │     └── invoke('get_environment_variables', { id })
  │
  └─ 5. 恢复完成 → 渲染界面
```

### 8.3 限制

- 临时请求 Tab 仅恢复 method + URL，不恢复未保存的 body/headers
- 最多恢复 20 个 Tab（超出部分丢弃）
- 响应数据不持久化（重启后需重新发送）
