## 一、发布阻断级问题（Critical）

### C1. 构建直接失败：图标缺失
- [ ] `tauri.conf.json:38-44` 引用 `32x32.png`、`128x128.png`、`128x128@2x.png`、`icon.icns`、`icon.ico` 共 5 个文件，`icons/` 目录实际只有 1 个 766 字节的占位 `icon.ico`。**`tauri build` 会直接报错。** 需要设计真实图标并生成全套尺寸（`pnpm tauri icon` 可从单张 1024x1024 源图生成）。

### C2. AI 聊天面板 XSS
- [ ] `AiChatPanel.tsx:236-238` 用 `dangerouslySetInnerHTML` 渲染消息：用户消息完全未转义直接注入 HTML；`renderMarkdown`（行 9-15）自写正则也不做 HTML 转义。AI 响应来自外部 API（不可信输入），`<img onerror=...>` 可在 Tauri webview 内执行。
- 修复：渲染前 escape HTML，markdown 渲染结果过 DOMPurify（或换用安全的 markdown 库）。

### C3. Vault 派生密钥写入 keyring — 与安全设计矛盾
- [ ] `crypto.rs:84-87`：`unlock_vault` 把 argon2 派生出的 **AES 主密钥 base64 后存入系统 keyring**。锁定 Vault 前，任何能读当前用户 keyring 的进程都能解密全部 secret，主密码形同虚设。且该 keyring 条目**只写不读**，是纯粹的泄漏面。
- [ ] `unlock_vault` **不验证密码正确性** — 输错密码也"解锁成功"，直到解密失败才报错。应在 vault 中存一个验证标记（如用主密钥加密的固定字符串），解锁时先验证。

### C4. Mock Server 绑定 0.0.0.0
- [ ] `mock.rs:171` 监听所有网卡，局域网内任何人都能访问用户的 mock 端点。应默认绑定 `127.0.0.1`，如需对外提供选项让用户显式开启。

### C5. Tauri 权限过宽：shell:allow-execute / allow-spawn
- [ ] `capabilities/default.json:18-21` 授予 webview **执行任意进程**的能力（`shell:allow-execute`、`allow-spawn`、`allow-stdin-write`、`allow-kill`）。结合 C2 的 XSS，攻击链可直达任意代码执行。除非确有需求，只保留 `shell:allow-open`。
- [ ] `tauri.conf.json:28` CSP `connect-src *` 放开了 webview 直连任意网络。架构要求所有请求走 Rust，CSP 应收紧为仅 ipc/localhost 作为兜底。

### C6. Cmd+W 关闭 Tab 绕过未保存确认，直接丢数据
- [ ] `App.tsx:55` 快捷键直接 `closeTab(activeTabId); removeTabData(activeTabId)`，而 TabBar 的关闭流程有 dirty 检查 + 三键确认对话框（`TabBar.tsx:35-41`）。快捷键关 Tab 时脏数据无提示丢失。
- 修复：抽取统一的 `closeTabWithConfirm` 入口，所有关闭路径（按钮、快捷键、右键菜单"关闭其他"等）都走它。

### C7. WS/MQTT/SSE 连接在 Tab 关闭后泄漏
- [ ] `removeConnection`（websocket-store.ts:109、mqtt-store.ts:140、sse-store.ts:89）**全项目无任何调用方**。Tab 关闭只调 `removeTabData`，Rust 侧连接保持打开、前端 `connections` Record 永久增长，消息继续写入已关闭 Tab 的 state。
- 修复：与 C6 一起做统一的 Tab 关闭清理入口：disconnect → removeConnection → removeTabData → console 清理。

### C8. 协议面板没有 UI 入口（功能不可达）
- [ ] `Workbench.tsx:81-99` 已实现 WebSocket/gRPC/MQTT/SSE/Mock 五个面板路由，但全代码库**没有任何地方以这些 protocol 打开 Tab**——侧边栏、命令面板、新建按钮都只能建 HTTP 请求。整套多协议功能用户摸不到。
- 修复：新建菜单/命令面板增加协议类型入口；或本版从构建中移除，避免死代码。

## 三、High 级问题

### Rust 后端

- [ ] **H1. SQLite schema 迁移有表无逻辑** — `storage/mod.rs` 建了 `schema_version` 表但从不读写。未来改表结构，老用户升级即数据损坏。1.0 前必须落地迁移执行逻辑。
- [ ] **H2. OAuth 1.0a / AWS v4 返回 NOT_IMPLEMENTED**（http.rs:339-344）— 对 API 客户端是核心功能缺口；UI 还允许选择并提示"凭证将不经签名发送"（messages.ts:738）。应实现或禁用入口。
- [ ] **H3. gRPC 实为 HTTP/1.1**（reqwest 而非 tonic，grpc.rs）— 无法对接标准 gRPC 服务器，功能名不副实。Cargo.toml 还声明了未使用的 tonic（徒增编译时间与供应链面）。
- [ ] **H4. pm.sendRequest 同步阻塞 + 线程泄漏** — engine.rs:176 使用 `reqwest::blocking`（超时 30s），而脚本超时机制是 `rx.recv_timeout`（engine.rs:77），**超时后线程不会被杀死**，只是不等结果：QuickJS runtime 继续后台跑。脚本死循环时线程永远不退出，反复触发积累僵尸线程。需 interrupt handler（`set_interrupt_handler`）。
- [ ] **H5. pm 注入存在 JS 注入风险** — `build_pm_js`（engine.rs:330-331,351）把 `req_url`、`req_method`、`resp_status_text` 用 `'{}'` 单引号直接拼进 JS 源码，未经 `serde_json::to_string` 转义。URL 含 `'` 即语法错误；`resp_status_text` 是服务端可控的，含注入 payload 即代码执行。统一改为 `{}`（用 `serde_json::to_string` 包一层）。
- [ ] **H6. Cookie 系统混乱与安全缺陷**：
  - reqwest `cookie_store(true)`（http.rs:255）与自建 SQLite cookie jar 并存，但每次请求新建 Client 导致 reqwest 内置 jar 实际无效。
  - `load_cookie_header`（http.rs:595）发送时不检查 expires/path/secure 属性，过期 cookie 永久发送。
  - `query_cookies`（storage/mod.rs:183）用 `LIKE` 模糊匹配域名，`a-example.com` 可能匹配到 `example.com` 的 cookie（跨域泄漏）。
- [ ] **H7. 每次请求新建 reqwest Client**（http.rs:367）— 无连接池复用，每个请求重复 TCP+TLS 握手，响应耗时测量失真。应按 settings 哈希缓存 Client（至少给 HttpClientState 加个 HashMap）。
- [ ] **H8. crypto.rs:139 `encrypt_vault_secret` name 未做路径校验** — `delete_vault_secret`（crypto.rs:191）校验了 `/`、`\`、`..`，但写入端遗漏。`name = "../../evil"` 可在 vault 目录外写文件。
- [ ] **H9. crypto.rs:69 salt 越界 panic** — `salt_bytes[..16]` 在 salt 文件 <16 字节时 panic（文件损坏/截断）。
