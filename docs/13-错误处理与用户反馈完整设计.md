## 1. 错误处理架构

### 1.1 错误流转链路

```
Rust 后端错误
  │
  ├─ AppError (Rust enum) ──── serialize ────► AppErrorResponse (JSON)
  │     { code, message, detail }                  │
  │                                                │ Tauri IPC
  │                                                ▼
  │                                          Frontend receives
  │                                          { code, message, detail }
  │                                                │
  │                                    ┌───────────┴───────────┐
  │                                    │  handleError(error)    │
  │                                    │  1. 分类 (by code)     │
  │                                    │  2. 严重度判断          │
  │                                    │  3. 展示 UI             │
  │                                    │  4. 日志记录            │
  │                                    └────────────────────────┘
  │
  └─ 前端原生错误 (JS Error)
        │
        └── ErrorBoundary / catch → handleError()
```

### 1.2 前端错误处理核心

```typescript
interface AppErrorResponse {
  code: string;
  message: string;
  detail: Record<string, unknown>;
}

enum ErrorSeverity {
  Info = 'info',
  Warn = 'warn',
  Error = 'error',
  Critical = 'critical',
}

function getSeverity(code: string): ErrorSeverity {
  const map: Record<string, ErrorSeverity> = {
    NET_REQUEST_CANCELLED: ErrorSeverity.Warn,
    VAULT_LOCKED: ErrorSeverity.Warn,
    VAULT_SECRET_NOT_FOUND: ErrorSeverity.Warn,
    AUTH_OAUTH_TOKEN_EXPIRED: ErrorSeverity.Warn,
    STORAGE_MIGRATION_FAILED: ErrorSeverity.Critical,
  };
  return map[code] ?? ErrorSeverity.Error;
}

function handleError(error: AppErrorResponse): void {
  const category = error.code.split('_')[0];
  const severity = getSeverity(error.code);

  switch (category) {
    case 'NET': showNetworkError(error); break;
    case 'SCRIPT': showScriptError(error); break;
    case 'VAULT': showVaultError(error); break;
    case 'IMPORT': showImportError(error); break;
    case 'STORAGE': showStorageError(error); break;
    case 'AUTH': showAuthError(error); break;
    default: showGenericError(error);
  }

  if (severity === ErrorSeverity.Critical) {
    logToSentry(error);
  }
}
```

### 3.2 逐错误码处理

| 错误码 | 展示位置 | 标题 | 消息格式 | 操作 |
|--------|---------|------|---------|------|
| `SCRIPT_PARSE_ERROR` | Console + 编辑器红色波浪线 | Syntax Error | `{scriptType} · Line {line}: {message}` | [Go to Line] 点击跳转编辑器 |

- 红色波浪线：`text-decoration: wavy underline var(--accent-danger)`
- Tooltip：错误消息 + 行号
- 行号区域：红色背景标记

## 11. 错误恢复策略（Recovery Strategy）

> docs-for-ai-dev D07 要求：每个错误码必须定义恢复策略（retry / fallback / degradation）。

### 11.1 网络错误恢复策略

| 错误码 | 恢复类型 | 恢复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| NET_CONNECT_FAILED | retry | 自动重试 1 次（间隔 1s），失败后提示用户 | ✅ 1次 | [Retry] 手动重试 |
| NET_TIMEOUT | retry + config | 建议增大 timeout_ms（默认 30000→60000） | ❌ | [Increase Timeout] [Retry] |
| NET_REQUEST_CANCELLED | none | 用户主动取消，无需恢复 | ❌ | [Retry] 重新发送 |
| NET_DNS_ERROR | retry | 自动重试 1 次（DNS 可能临时故障） | ✅ 1次 | [Check URL] [Retry] |
| NET_TLS_ERROR | fallback | 前端提供 "Skip SSL Verify" 按钮，降级为 verify_ssl=false | ❌ | [Skip SSL Verify] [Retry] |
| NET_SSL_VERIFICATION | fallback | 同 NET_TLS_ERROR | ❌ | [Skip SSL Verify] |
| NET_REDIRECT_LIMIT | config | 建议增大 max_redirects 或关闭 follow_redirects | ❌ | [Disable Follow Redirects] |
| NET_BODY_TOO_LARGE | degradation | 截断显示前 1MB + 提供下载选项 | ❌ | [Download as File] |
| NET_INVALID_URL | validation | 前端拦截（不发送到 Rust），URL 输入框红色标记 | ❌ | 修正 URL |
| NET_AUTH_FAILED | retry | 建议检查 Auth 配置，特别是 token 过期 | ❌ | [Check Auth] [Retry] |

### 11.2 存储错误恢复策略

| 错误码 | 恸复类型 | 恸复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| STORAGE_READ_FAILED | retry | 重试 1 次（文件可能被临时锁定） | ✅ 1次 | [Retry] |
| STORAGE_WRITE_FAILED | fallback | 先写 .tmp 再 rename（原子写入），失败后提示 Save As | ❌ | [Save As...] [Retry] |
| STORAGE_NOT_FOUND | fallback | 创建默认数据结构（空集合/空环境） | ✅ | — |
| STORAGE_PATH_TRAVERSAL | none | 拒绝操作（安全策略，不恢复） | ❌ | — (安全拦截) |
| STORAGE_PARSE_FAILED | fallback | 从 .bak 文件恢复 | ✅ | [Restore Backup] |
| STORAGE_FILE_TOO_LARGE | degradation | 拒绝写入 + 提示文件大小限制 | ❌ | [Reduce File Size] |

### 11.3 脚本错误恢复策略

| 错误码 | 恢复类型 | 恢复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| SCRIPT_TIMEOUT | config | 建议增大超时（默认 5s→10s） | ❌ | [Increase Timeout] |
| SCRIPT_ERROR | none | 显示错误详情 + 行号，用户修改脚本 | ❌ | [Go to Line] |
| SCRIPT_MEMORY_LIMIT | degradation | 强制终止脚本，不重试 | ❌ | [Optimize Script] |

### 11.4 Vault 错误恢复策略

| 错误码 | 恢复类型 | 恢复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| VAULT_LOCKED | action | 提示解锁 Vault（模态框） | ❌ | [Unlock Vault] |
| VAULT_UNLOCK_FAILED | retry | 清空密码框 + 提示重新输入 | ❌ | 重新输入密码 |
| VAULT_ENCRYPT_FAILED | retry | 重试加密操作 | ✅ 1次 | [Retry] |
| VAULT_DECRYPT_FAILED | fallback | 提示密码可能错误，建议重新解锁 | ❌ | [Re-enter Password] |

### 11.5 验证错误恢复策略

| 错误码 | 恸复类型 | 恸复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| VALIDATION_FAILED | validation | 高亮错误字段 + 显示错误消息 | ❌ | 修正输入 |
| INVALID_INPUT | validation | 同 VALIDATION_FAILED | ❌ | 修正输入 |

### 11.6 代理错误恢复策略

| 错误码 | 恸复类型 | 恸复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| PROXY_START_FAILED | retry | 重试启动代理 | ❌ | [Check Proxy Config] [Retry] |
| PROXY_NOT_RUNNING | action | 提示启动代理 | ❌ | [Start Proxy] |

### 11.7 通用错误恢复策略

| 错误码 | 恸复类型 | 恸复策略 | 自动恢复 | 用户操作 |
|--------|---------|---------|---------|---------|
| NOT_IMPLEMENTED | none | 提示功能尚未实现 | ❌ | — |
| INTERNAL | retry | 自动重试 1 次，失败后显示详情 | ✅ 1次 | [Report Issue] |
